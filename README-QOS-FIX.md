# پچ QoS/NetLimiter — رفع تداخل SmartLaunch/UVNC/PsExec

## ریشه‌ی مشکل (که توی کد پیدا شد)

1. **مسیر ری‌اپلای بعد از UVNC واقعاً خراب بود**: `exir-client-agent.mjs`
   (روی هر VIP، پورت 8766) هنوز از `nlq.exe` و رول‌های نام‌دار `Exir-500K`
   استفاده می‌کرد — همون چیزی که خود کامنت‌های `ping-agent.mjs` گفته بودن
   توی NetLimiter 4.1.13 وجود نداره. یعنی هر بار UVNC باز می‌شد و تلاش
   خودکار برای برگردوندن QoS انجام می‌شد، این تلاش بی‌صدا شکست می‌خورد.
2. **کلیک مستقیم روی تیر (500K و...) به PsExec وابسته بود** که با یک
   Session باز SMB از طرف SmartLaunch/UVNC به همون هاست تداخل می‌کرد
   (خطای ویندوز 5 "Access is denied" یا 1219 "Multiple connections...").
   `clearSmbSession` قبلاً هم بود ولی فقط یک بار، بدون retry.
3. **هیچ خطایی به UI نشون داده نمی‌شد** — `pushQos` نتیجه واقعی رو چک
   نمی‌کرد، پس حتی اگه هر دو بالا شکست می‌خوردن، داشبورد ساکت می‌موند.

## چی تغییر کرد

### `Files/exir-client-agent/exir-client-agent.mjs`
- به‌جای `nlq.exe`/رول‌های نام‌دار، حالا مستقیم `netlimiter-qos.ps1` (همون
  اسکریپتی که با DLL reflection کار می‌کنه و از قبل تست‌شده بود) رو محلی
  اجرا می‌کنه. چون این agent از قبل داخل Session تعاملی روی خود VIP در حال
  اجراست، اصلاً نیازی به SMB/PsExec نداره → دیگه با SmartLaunch/UVNC تداخل
  Session معنی نداره.
- روت `/netlimiter/apply` حالا `{ tier, bytes }` می‌گیره (مقدار بایت از
  سرور مرکزی محاسبه و پاس داده می‌شه، همون‌طور که برای PsExec هم بود).

### `ping-agent.mjs`
- `handleQosNetLimiter` حالا اول از client-agent (پورت 8766) استفاده
  می‌کنه (بدون SMB، بدون تداخل). فقط اگه client-agent در دسترس نبود
  (مثلاً هنوز نصب نشده)، میره سراغ PsExec.
- مسیر PsExec هم بهبود پیدا کرد: `clearSmbSession` قبل از هر تلاش خودکار
  اجرا می‌شه، و اگه خطا شبیه تداخل Session/Logon بود (کد 5 / 1219 / 1385)،
  خودش یک بار دیگه پاکسازی می‌کنه، کمی صبر می‌کنه، و دوباره تلاش می‌کنه —
  دقیقاً همون چیزی که خواستید: "خودش پاکسازی رو انجام بده و بعد دستور
  تنظیم QoS رو ارسال کنه."
- روت `/netlimiter/reapply` (که بعد از باز شدن UVNC صدا زده می‌شه) هم
  اصلاح شد تا مقدار بایت رو مثل مسیر اصلی محاسبه کنه.

### `src/lib/qos.ts` و `src/components/monitoring/QosLaunch.tsx`
- `pushQos` حالا نتیجه واقعی (`ok`/`error`) رو برمی‌گردونه، نه فقط
  موفقیت HTTP request.
- اگه اعمال روی کلاینت واقعاً شکست بخوره، زیر همون دکمه‌ی تیر یک پیام
  خطای قرمز نشون داده می‌شه (به‌جای سکوت کامل قبلی).

## قدم بعدی برای دیپلوی

1. فایل `exir-client-agent.mjs` جدید رو جایگزین نسخه‌ی روی هر VIP کنید
   (همون مسیری که `install-service.ps1` نصب کرده) و agent رو ری‌استارت
   کنید (لاگ‌اوت/لاگین یا ری‌استارت scheduled task کافیه).
2. مطمئن بشید `netlimiter-qos.ps1` روی هر VIP سر جاشه (اگه
   `Setup-NetLimiter-VIP.ps1` رو قبلاً اجرا کرده باشید، هست).
3. `ping-agent.mjs` جدید رو روی سرور مرکزی جایگزین کنید و ری‌استارتش کنید.
4. فرانت‌اند رو rebuild کنید (`qos.ts` و `QosLaunch.tsx` جدید).
5. تست: با UVNC باز، روی یک VIP کلیک کنید و تیر رو عوض کنید — الان باید
   یا از client-agent (بدون خطا) اعمال بشه، یا اگه client-agent نصب
   نبود، PsExec با auto-cleanup دوباره تلاش کنه؛ در صورت شکست واقعی هم
   پیام خطا زیر دکمه دیده می‌شه (به‌جای سکوت).

## نکته‌ی جانبی
دستور دستی PsExec که فرستادید، خطاش فقط PATH بود (`PsExec.exe` رو از
مسیری اجرا کنید که در `PATH` باشه، یا مسیر کامل بدید — مثل
`Files\PSTools\PsExec.exe` یا ریشه‌ی پروژه)؛ ربطی به باگ اصلی نداشت.
